# V10.13r — Cold-Start Recovery Patch (Post-DB-Wipe Entry Unblock)

## Role
You are a senior quant/backend engineer working directly in the existing CryptoMaster Python codebase.
Your task is to implement a **safe cold-start recovery patch** for a live trading bot that was recently reset (DB wiped + bot restarted).

The problem is **not** that the bot is broken. The pipeline is alive, but after reset it is still too restrictive during bootstrap.

Current live symptom:
- After DB wipe and restart, bot made only **1 trade in 20 minutes**
- This is too low for expected live activity
- It indicates **entry deadlock / over-filtering during cold start**

Your goal is to **increase safe trade throughput during early bootstrap** without removing core protections.

---

## Confirmed Situation

### What is already true
- Pipeline is alive
- Signals are generated
- At least some trades can still execute
- Deadlock is partial, not total
- Previous V10.13q reduced extreme blocking significantly

### What logs showed before
- Pre-live audit improved from catastrophic blocking to roughly acceptable range
- But live cycles still often show:
  - `Signaly (THIS CYCLE) 0 kandidati`
  - very low real executed trade frequency
- Main blockers observed across recent logs:
  - `OFI_TOXIC_HARD`
  - `FAST_FAIL_HARD`
  - `FREQ_CAP`
  - `SKIP_SCORE_SOFT`
- After wipe DB, learning data is sparse, so strict mature filters are likely firing too early

### Important interpretation
This is **not** a request to globally loosen the bot.
This is a request to make the bot **bootstrap-tolerant** and then automatically return to normal stricter behavior.

---

## Objective

Implement a **Cold-Start Recovery Mode** that activates automatically after reset / sparse history and only affects the early bootstrap window.

### High-level target
Make the bot move from:
- ~1 trade / 20 minutes

to something more like:
- **5+ trades per hour** in a normal active market
- while preserving:
  - EV gate logic
  - risk manager
  - execution quality guard
  - OFI toxicity protection for truly catastrophic conditions
  - portfolio exposure constraints

---

## Required Design

## 1. Add explicit cold-start mode
Create a deterministic helper in the decision layer, for example in `realtime_decision_engine.py`:

- `is_cold_start(...)`
- or `get_bootstrap_state(...)`

It should return `True` when the bot is in sparse-history bootstrap.

Use robust conditions, for example combining:
- low global completed trades
- low pair/regime sample count
- recent DB reset style conditions
- optional time-since-start fallback

### Suggested activation logic
Use something like this conceptually:
- global completed trades `< 100`
OR
- pair/regime sample count `< 15`
OR
- uptime since restart `< 60 minutes` **and** total completed trades still low

Do **not** depend on only one fragile signal.

### Required behavior
Cold-start mode must be:
- automatic
- deterministic
- visible in logs/dashboard
- easy to disable later

---

## 2. Soften OFI hard blocks during cold start
Current problem:
- `OFI_TOXIC_HARD` is likely killing too many candidates too early

Required patch:
- Preserve **true catastrophic OFI block**
- But downgrade borderline OFI failures from `HARD` to `SOFT` while in cold-start mode

### Required behavior
In cold start:
- catastrophic flow contradiction still hard-blocks
- borderline toxic flow only penalizes confidence / score / auditor factor
- does not instantly kill otherwise decent bootstrap candidates

### Safety requirement
Do **not** allow entries against obviously toxic flow extremes.

---

## 3. Convert FAST_FAIL_HARD into bootstrap-soft behavior for low sample sizes
Current problem:
- after DB wipe, `FAST_FAIL_HARD` is using immature data too aggressively

Required patch:
- When pair/regime sample size is still immature, hard fail should become soft fail
- Apply penalty, not kill
- Keep hard fail only for mature sample size or clearly catastrophic cases

### Suggested rule
If `n < mature_n_threshold`:
- FAST_FAIL_HARD => FAST_FAIL_SOFT_BOOTSTRAP
- reduce confidence / auditor factor / score
- allow candidate to continue if remaining gates still pass

If `n >= mature_n_threshold`:
- keep current stricter logic

---

## 4. Relax frequency cap during bootstrap
Current problem:
- `FREQ_CAP` may be unnecessarily suppressing recovery after reset

Required patch:
- In cold-start mode, make frequency cap adaptive and slightly looser
- especially when:
  - no open positions
  - recent inactivity exists
  - completed trades still low

### Required behavior
Cold-start should not allow spam or overtrading.
But it should stop one recent trade from suppressing the next 10–20 minutes of exploration.

A good approach:
- keep cooldown
- reduce cooldown during cold start
- reduce it more when no positions are open and idle time is growing

---

## 5. Slightly relax entry thresholds only in bootstrap
Current problem:
- mature thresholds appear too strict for a freshly reset bot

Required patch:
Only in cold-start mode, slightly relax:
- EV threshold
- score threshold
- maybe weighted score threshold if present

### Guardrails
This must be **small**, not reckless.

Example intent:
- EV threshold mildly reduced
- score threshold mildly reduced
- never bypass risk or execution quality
- never allow random low-quality noise trades

---

### Also log per-cycle summary where useful

This is critical. No blind patch.

---

## 7. Automatic return to mature mode
Cold-start mode must turn off automatically.

### Exit conditions
For example:
- global completed trades >= 100
AND
- enough pair/regime maturity exists
OR
- uptime exceeds fallback window and trade history is healthy

When cold-start ends:
- revert to standard mature behavior
- no manual toggle required

---

## 8. Do not break existing safety architecture
Must preserve:
- risk budgeting
- max position logic
- exposure caps
- execution quality checks
- cost guard / net edge guard
- OFI catastrophic hard blocks
- emergency override semantics
- existing telemetry formatting as much as possible
