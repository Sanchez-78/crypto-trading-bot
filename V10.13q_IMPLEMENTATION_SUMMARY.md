## OVERVIEW

V10.13q is a **production-critical entry pipeline unclog patch**. The bot is **profitable and stable** (56.8% WR, 1.84x profit factor) but was suffering from **complete entry deadlock**: almost no new trades executed due to over-aggressive pair/regime blocking.

**Root cause**: `PAIR_BLOCK` hard kills were rejecting 312k+ candidates. `FAST_FAIL_SOFT` was behaving like a shadow hard filter (114k rejections). Emergency recovery couldn't override pair blocks.

**Solution**: Convert hard blocks to **graduated penalties**, allow **emergency override**, and **never hard-kill on soft blockers**.

#### Change 1C: PAIR_BLOCK logic rewritten (V10.13q)
**Lines 1053-1124** — Pair/regime block evaluation

**What changed**:
Old logic (hard kill):
```python
if (_pn >= 15 and _pwr < 0.10) or (_pn >= 25 and _pwr < 0.30):
    return None  # ← HARD KILL
```

New logic (staged penalties):
```python
# Tier 4: n≥30, WR<10%  → 20% size penalty (80% reduction) + score+0.05
# Tier 3: n≥25, WR<20%  → 25% size penalty + score+0.02
# Tier 2: n≥20, WR<30%  → 50% size penalty + ev+0.01
# Tier 1: n≥15, WR<40%  → 75% size penalty

# Emergency override: if in recovery + idle + 0 positions → allow Tier 4, soften Tier 3
```

**Variables introduced**:
- `_pair_penalty = 1.0` (size multiplier)
- `_pair_ev_uplift = 0.0` (EV threshold increase)
- `_pair_score_uplift = 0.0` (score threshold increase)
- `_pair_hard_blocked = False` (terminal decision)
- `_pair_block_reason = ""` (diagnostics)

**Why**:
- Root cause A: `SKIP_PAIR` was a global entry choke point (examples: BTCUSDT/BEAR 29% WR, ETHUSDT/BEAR 24% WR)
- Root cause B: Emergency recovery couldn't override pair blocks
- Staged penalties preserve safety (reduces size first, requires higher EV/score) but allow exploratory trades

**Safety**: ✅ Tier 4 still hard blocks unless recovery active; penalties have multiplicative bounds; all three layers (size, EV, score) are hit

#### Change 1E: FAST_FAIL_SOFT reworked to be truly soft
**Lines 1015-1051** — FAST_FAIL logic

**Old behavior** (implicit hard kill):
```python
if _ff_wr < 0.20 and _ff_ev <= 0.0:
    _fast_fail_soft = True
    # Penalties applied...
    track_blocked(reason="FAST_FAIL_SOFT")
    return None  # ← STILL RETURNED NONE (killed it)
```

**New behavior** (truly soft, no kill):
```python
if _ff_wr < 0.20 and _ff_ev <= 0.0:
    _fast_fail_soft = True
    _ff_penalty = max(0.30, 0.60 - (_ff_wr * 3.0))
    # Penalties applied but NO return None
    # ← Allows candidate through with reduced confidence
    track_blocked(reason="FAST_FAIL_SOFT")  # Diagnostic only
    # ... continue to score gate ...
```

**Why**:
- Root cause C: `FAST_FAIL_SOFT` had massive count (114k) suggesting it was behaving like a hard filter
- Penalty is now gradient: WR 19% → 0.60x confidence, WR 10% → 0.45x, WR 5% → 0.30x
- Allows borderline pairs to attempt trade if other conditions favor it

**Safety**: ✅ Confidence reduction means EV gate is harder to pass; penalty is multiplicative (can't boost)

## KNOWN RISKS & MITIGATIONS

| Risk | Mitigation |

| Tier 1/2 penalties too lenient → more bad trades | Kill audit will show immediately if PAIR_BLOCK allows losers through |
| Emergency override exploited in normal markets | Override requires idle>threshold + 0-positions + recovery active (triple gate) |
