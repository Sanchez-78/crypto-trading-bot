# V10.13o — Scratch Rebalance Patch

**Status:** Proposed  
**Type:** Safety / performance tuning patch  
**Priority:** High  
**Goal:** Reduce premature scratch exits without reintroducing timeout dominance

---

## Executive Summary

V10.13n fixed the timeout problem successfully, but the exit mix has shifted too far toward `SCRATCH_EXIT`.

Observed live behavior after V10.13n:
- `timeout` exits dropped from meaningful levels to near zero
- `PARTIAL_TP_25` started firing, which is positive
- `TRAILING_STOP` remained weak
- `SCRATCH_EXIT` became dominant (`~79%` of exits in the observed window)
- `PARTIAL_TP_50` and `PARTIAL_TP_75` remained at zero
- Recent 50-trade winrate degraded versus the longer baseline

This strongly suggests that positions are being released too early via scratch logic before they can mature into higher-value exits.

---

## Diagnosis

### What is working
- Timeout suppression is working
- Exit counters are working
- Partial harvest path is finally alive
- Bot remains stable and profitable
- No runtime faults observed

### What is not working
- Scratch has become the default exit rather than a fallback release
- Higher-order profit milestones (`PARTIAL_TP_50`, `PARTIAL_TP_75`) are never reached
- Trailing has not improved enough to justify the new earlier release behavior
- Recent live trade quality softened (`last 50 trades` worse than full-sample average)

### Core hypothesis
`SCRATCH_EXIT` is currently too permissive on both:
1. **time dimension** — activates too early  
2. **PnL band dimension** — allows too wide a flat-profit / flat-loss release band

As a result, trades are being closed before they can either:
- develop into `PARTIAL_TP_25`
- develop into `PARTIAL_TP_50`
- develop into `PARTIAL_TP_75`
- arm and realize `TRAILING_STOP`

---

## Patch Objective

Rebalance exit sequencing so that:
- `SCRATCH_EXIT` remains available as a capital-release tool
- but stops dominating the exit distribution
- while preserving the timeout improvements achieved by V10.13n

---

## Proposed Changes

## 1. Make scratch older before it can fire

### Change
```python
SCRATCH_MIN_AGE_S = 105   # or 110
```

### Current
```python
SCRATCH_MIN_AGE_S = 85
```

### Rationale
At `85s`, scratch activates too early and intercepts trades before they can evolve into:
- `PARTIAL_TP_25`
- `PARTIAL_TP_50`
- `PARTIAL_TP_75`
- `TRAILING_STOP`

Moving scratch activation later gives trades more room to develop while still keeping it earlier than historical timeout behavior.

### Recommended value
**Primary recommendation:** `105s`  
**More conservative alternative:** `110s`

---

## 2. Narrow the scratch PnL band

### Change
```python
SCRATCH_MAX_PNL = 0.0016   # or 0.0017
```

### Current
```python
SCRATCH_MAX_PNL = 0.0020
```

### Rationale
The current band is likely too wide. That means trades that are still “alive” and capable of maturing are being classified as flat-enough and released.

Reducing the allowed scratch band should:
- reduce scratch dominance
- allow more trades to reach partial take-profit milestones
- give trailing logic more chance to engage

### Recommended value
**Primary recommendation:** `0.0016`  
**Softer alternative:** `0.0017`

---

## 3. Keep partial TP thresholds unchanged for now

### Keep
```python
PARTIAL_TP_25_THRESHOLD = 0.15
PARTIAL_TP_50_THRESHOLD = 0.35
PARTIAL_TP_75_THRESHOLD = 0.60
```

### Rationale
`PARTIAL_TP_25` is already firing, which proves the path works.
The absence of `50/75` is more likely caused by premature scratch exits than by bad thresholds.

Changing both scratch and partial thresholds at the same time would make diagnosis harder.

**Conclusion:** keep partial TP milestones unchanged in V10.13o.

---

## 4. Do not change trailing activation yet

### Keep
```python
TRAILING_ACTIVATION = 0.0020
```

### Rationale
Trailing is underperforming, but the current evidence suggests it may be starved by premature scratch exits rather than intrinsically misconfigured.

If scratch releases positions before they survive long enough to arm trailing logic, then changing trailing first would attack the wrong layer.

**Conclusion:** leave trailing unchanged in V10.13o.

---

## 5. Add pair-level caution for ADA

### Problem
ADA remains a persistent drag across multiple checkpoints:
- weak winrate
- negative contribution
- weaker quality than BTC/ETH/BNB/DOT

### Minimal response
Apply a lightweight pair-level restriction instead of changing global logic.

### Options

#### Option A — size penalty
```python
if symbol == "ADAUSDT" and pair_wr < 0.45:
    size_mult *= 0.75
```

#### Option B — temporary cooldown bias
```python
if symbol == "ADAUSDT" and pair_ev <= 0:
    require_extra_score += 0.01
```

#### Option C — exploration suppression
```python
if symbol == "ADAUSDT" and recent_loss_cluster:
    suppress_forced_explore = True
```

### Recommendation
Use **Option A** first because it is the least invasive and does not completely disable learning.

## Why this patch is intentionally small

This patch should stay narrow because:
- V10.13n already fixed a real structural problem
- timeout elimination is a valid success
- the current issue is **distribution skew**, not total failure
- changing too many thresholds at once would destroy observability

V10.13o should therefore be a **rebalancing patch**, not a redesign.

---

## Final Recommendation

**Proceed with V10.13o as a minimal corrective patch.**

### Summary of action
1. Raise `SCRATCH_MIN_AGE_S`
2. Narrow `SCRATCH_MAX_PNL`
3. Leave partial and trailing thresholds unchanged
4. Optionally add light ADA size penalty
5. Reobserve exit mix for 50–100 exits

This is the most evidence-aligned next step based on the live data collected so far.
