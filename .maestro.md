## Workflow Architecture

**Typical Workflow Loop** (10 steps):

1. User describes problem, pastes logs/screenshots, or points to narrow codebase area
2. Claude Code analyzes in **read-only mode** → identify root cause, affected files, risks, minimal patch scope
3. Claude Code creates patch plan; if coding needed, prepare scoped implementation task for Codex
4. Codex implements small, mechanical, file-scoped changes where possible
5. Claude Code reviews diff for:
   - Logic regressions
   - Architecture breakage
   - Metric consistency
   - Firebase budget safety
   - Production risk
6. Tests/checks run locally first
7. Pre-live audit run before deployment
8. Changes committed only after validation
9. Deployment to Hetzner/Render/Railway verified by:
   - Production logs (systemd/Render/Railway)
   - Active version markers
   - Expected runtime behavior
10. Live monitoring confirms no regression

**Key Tools**:
- Grep/search and targeted file reads (never grep the whole repo without reason)
- Production logs (Hetzner systemd, Render/Railway dashboards)
- Git diff and git history (baseline and rollback reference)
- Local Python compile/import checks
- Unit/smoke tests (where available)
- Pre-live audit (regression gate)
- Firebase metrics and read/write budget tracking
- Android dashboard consistency checks
- Runtime version markers in logs

**No Full Rewrites**: Analysis must precede every patch. Changes are incremental, scoped, and minimal. Never rewrite architecture unless explicitly requested.

## Quality & Evaluation

### Definition of Done (All Must Be Met)

- ✅ Code compiles/imports without errors
- ✅ Targeted tests or smoke checks pass
- ✅ Pre-live audit shows no critical regression
- ✅ Canonical metrics remain consistent
- ✅ EV/RDE semantics preserved (unless task explicitly changes them)
- ✅ Firebase read/write budget not worsened
- ✅ Production logs show expected version marker and expected behavior
- ✅ No important debug logs, metrics, or Android dashboard fields disappear
- ✅ Rollback plan documented
- ✅ For live trading changes: bot runs observation window without obvious regression

### Regression Definition

**Hard Regressions** (unacceptable):
- Bot stops generating signals or executing valid decisions
- Existing production logs, debug diagnostics, or version markers disappear
- EV/RDE semantics change silently
- Canonical metrics inconsistent between backend/audit/Firebase/Android
- Dashboard shows different values than `canonical_metrics.py`
- TIMEOUT/TP/SL/scratch exits misclassified
- Firebase reads exceed 50k/day
- Firebase writes exceed budget or increase materially without approval
- Code path writes excessive per-tick/per-price data
- Production deployment succeeds but runtime still shows old version
- Error handling hides failures instead of logging actionable diagnostics
- Any full rewrite removes working features
- Trading behavior changes without validation and rollback plan

**Soft Regressions / Warning Thresholds**:
- Profit factor drops >0.05 without explanation
- Win rate drops >3pp without explanation
- Expectancy/EV trend worsens materially
- Timeout ratio increases materially
- `forced_blocked_ratio` changes significantly without explanation
- Signal count drops materially (no clear market/regime reason)
- Firebase read/write volume increases >10%
- Log noise increases, obscuring important diagnostics
- Android dashboard latency or freshness worsens

### Golden Examples & Test Scenarios

Use **rolling baselines** — do not assume one permanent baseline because trading data/regimes change:
- Last known good `pre_live_audit` output
- Last known good `canonical_metrics` snapshot
- Last known good production log sample
- Last known good Firebase metrics snapshot
- Last known good Android dashboard view
- Git commit immediately before patch (rollback baseline)
- Synthetic scenarios: TP, SL, TIMEOUT, scratch, rejected signal, accepted signal, Firebase write compression, empty/cold-start DB

**Comparison Style**:
- Compare before vs after
- Explain whether difference is expected
- If metrics change, identify exact code path and reason
- Never claim improvement from one metric only
- Validate with logs, tests, and canonical metrics together

### Common Failure Modes (Watch For)

1. **Silent behavior changes** in gates: EV/RDE, score threshold, timing, pair block, fast fail, loss cluster, velocity, streak, emergency gates
2. **Metric inconsistencies**: dashboard ≠ audit ≠ Firebase ≠ `canonical_metrics.py`
3. **Exit misclassification**: TIMEOUT marked as win despite negative/zero PnL
4. **PnL rounding**: too aggressive, hiding small losses/fees
5. **Fees not in net_pnl**: exit profitability wrong
6. **Redis failure**: repeated noisy logs or blocking logic
7. **Firebase quota surprise**: unbounded reads/writes
8. **Deployment mismatch**: code compiled but not actually deployed
9. **Wrong process on Hetzner**: version mismatch between git and runtime
10. **Cleanup casualties**: debug logs or metrics removed while "optimizing"
11. **Overengineering**: too many meta layers, guards, and penalties → bot stops trading
12. **Under-validation**: compiles but live behavior not verified
13. **Android consistency**: reading inconsistent Firestore documents
14. **Bootstrap edge cases**: regression tests miss cold-start behavior
15. **Silent EV/RDE changes**: gates adjusted without documenting reasoning

## Constraints

### Cost / Budget

**Firebase**:
- Reads: stay under 50k/day (hard limit)
- Writes: stay controlled; avoid per-tick writes, prefer compression/batching
- Budget impact: measure before/after every change

**Claude/Codex Tokens**:
- Minimize token usage via targeted file reads, grep/search, summaries, scoped prompts
- Do NOT load whole repository unless truly necessary
- Prefer Codex for scoped implementation; Claude for architecture/review
- Write durable summaries into `.md` files to avoid repeated large prompts

**Deployment**:
- Controlled frequency; do not deploy every minor idea without validation
- Each deployment costs time, attention, and rollback readiness

### Latency

- **Production-breaking issues**: analyze and patch as fast as safely possible
- **Normal improvements**: quality and safety > speed
- **Patches**: incremental and independently deployable
- **Live trading patches**: require validation before and after deployment
- **Low confidence**: propose diagnostic instrumentation instead of risky logic changes

### Compliance / Safety

**Critical** — real-money trading context:
- No change may silently increase risk, leverage, exposure, order frequency, or trade size
- Every trading logic change must include: expected behavior, validation commands, log evidence, rollback plan
- Sensitive data (API keys, Firebase service accounts, tokens, passwords, private keys):
  - Never print, commit, summarize into prompts, or expose
  - Logs/metrics allowed for debugging; sanitize if they contain secrets
- Every production patch must have audit trail:
  - Git diff
  - Commit message
  - Validation result
  - Deployment confirmation
  - Rollback command
- Prefer fail-safe over hidden failure
- Never hide errors that affect trading decisions, execution, persistence, metrics

## Priorities (Ranked 1–5)

1. **Safety** — production risk mitigation, rollback readiness, no dangerous live-trading behavior
2. **Quality** — correctness, no regressions, canonical metrics consistency, validated behavior
3. **Explainability** — understand why a change works and what it affects
4. **Cost** — Claude/Codex tokens, Firebase quota, compute, log noise
5. **Speed** — useful, but never at expense of safety or correctness

### Single Biggest Problem to Solve

**Prevent AI-driven regression and chaos in a large live trading bot.**

Maestro should enforce a disciplined workflow where:
- Claude/Codex first understand existing architecture
- Preserve working logic
- Make only scoped incremental changes
- Validate with evidence
- Never remove metrics/logs/features
- Never silently change EV/RDE/execution behavior
- Maintain strict audit trail for production changes

## Workflow Rules (Non-Negotiable)

1. **Read-only analysis first** — always
2. **No full rewrites** — unless explicitly requested
3. **No silent behavior changes** — all EV/RDE/timing changes are explicit
4. **No removing metrics/logging** — ever
5. **Prefer minimal incremental patches** — over comprehensive refactors
6. **Always state affected files and validation commands**
7. **If confidence is low, stop and ask** for logs or propose diagnostic patch instead of guessing
8. **Rollback plan required** for every production change
9. **Token optimization required** — use grep, targeted reads, summaries; no full-repo prompts
10. **Durable artifacts** — summarize important findings into `.md` files for reuse
