ISSUE 1: LATENCY CRISIS (270ms avg vs 50ms SLA)
================================================================================

ROOT CAUSE:
- Firebase batch.commit() is SYNCHRONOUS and blocking
- Called from trade close path (~2200ms in trade_executor.py)
- Every trade close waits 200-300ms for Firebase round-trip

LIVE IMPACT:
All 6 observed latencies at 5-6x SLA:
  05:38 — 325.78ms
  05:40 — 256.92ms
  05:43 — 263.89ms
  05:45 — 259.80ms
  05:46 — 299.62ms
  05:52 — 224.12ms
  Average: ~272ms (5.4x over limit)

FIX APPLIED (src/services/firebase_client.py):
- New _async_firebase_write() function (background thread executor)
- save_batch() now queues commit to thread, returns immediately
- Local cache updated synchronously for consistency
- No change to external API or data integrity

BEFORE:
  trade close → save_batch() → fb_batch.commit() [BLOCKS 200-300ms] → return

AFTER:
  trade close → save_batch() [updates cache] → spawn thread [commit async] → return [<5ms]

EXPECTED RESULT:
- Trade close latency: ~270ms → <50ms (95% reduction)
- Market stream remains responsive to price ticks
- Firebase writes still complete, just asynchronously

VERIFICATION:
After deployment, monitor latency in live logs for next 50+ trades:
  - Should see LATENCY_WARN values drop from 200-300ms to <50ms
  - [FIREBASE_WRITE] messages should show "async write, non-blocking"
