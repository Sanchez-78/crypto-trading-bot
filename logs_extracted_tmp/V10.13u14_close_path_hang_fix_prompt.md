# V10.13u+14 — Close Path Hang Fix / Non-Blocking Close Reconciliation

## Context
Production after V10.13u+13 confirms the force-reconcile fallback works, but close execution still hangs repeatedly:

Observed:
```text
[CLOSE_SKIP_DUPLICATE] BNBUSDT ... status=already_closing age=11s attempts=37
[CLOSE_LOCK_STALE_RELEASE] ... count=1
...
[CLOSE_LOCK_STALE_RELEASE] ... count=3
[CLOSE_FORCE_RECONCILE] ... changed=True
```

Affected exit types:
- `SCRATCH_EXIT`
- `replaced`
- `PARTIAL_TP_25`

This means the lock/reconcile layer is working as an emergency safety net, but the real close path is still not reaching normal completion/removal within 20s.

## Current Status
Do NOT reimplement V10.13u+8–u+13.

Known-good:
- Runtime version/commit injection works.
- Economic PF is canonical and safe.
- Exit PnL integrity is mostly fixed.
- Duplicate close storms are suppressed.
- Stale locks are released.
- Force reconcile removes stuck positions.

Still broken:
- Close path can hang before normal close completion.
- Positions are often removed only by force reconcile, not by normal close flow.
- `PARTIAL_TP_25` is incorrectly treated like a full close lock in some paths.
- Logs do not show enough stage-level evidence to identify exact hanging step.

## Goal
Fix the root cause of close hangs, not just the symptom.

Success means:
- Most closes complete normally with `[CLOSE_LOCK_RELEASED] ... status=closed`.
- `[CLOSE_FORCE_RECONCILE]` becomes rare/emergency only.
- `PARTIAL_TP_25` does not fully remove/reconcile a still-open partial position.
- No infinite `already_closing`.
- No `EXIT_INTEGRITY_ERROR`.
- No change to PF formula, EV-only logic, Firebase quotas, or entry strategy.

## Patch Requirements

### 1. Add Stage-Based Close Tracing
In `trade_executor.py`, add low-spam diagnostic stages around close flow.

Use a helper:
```python
def _close_stage(sym: str, key: str, stage: str, **meta) -> None:
    log.warning(
        "[CLOSE_STAGE] symbol=%s key=%s stage=%s %s",
        sym, key, stage,
        " ".join(f"{k}={v}" for k, v in meta.items())
    )
```

Emit stages:
- `lock_acquired`
- `pnl_computed`
- `trade_record_built`
- `notifier_start`
- `notifier_done`
- `metrics_start`
- `metrics_done`
- `firebase_start`
- `firebase_done`
- `position_remove_start`
- `position_remove_done`
- `lock_release_start`
- `lock_release_done`
- `exception`

Throttle if needed, but do not hide exceptions.

### 2. Wrap Close Body in Safe Try/Except/Finally
The previous patch avoided try/finally because of indentation risk. Now do a surgical extraction instead.

Refactor:
```python
def close_position(...):
    acquired, key, status = _try_acquire_close_lock(...)
    if not acquired:
        return None
    try:
        return _close_position_inner(..., close_key=key, close_lock_status=status)
    except Exception as e:
        log.exception("[CLOSE_EXCEPTION] symbol=%s key=%s reason=%s err=%r", sym, key, reason, e)
        raise
    finally:
        _release_close_lock(key, sym, reason, status="closed_or_exception")
```

Move the existing close implementation into `_close_position_inner()` with minimal edits.

Acceptance:
- Lock release must happen even if notifier, Firebase, attribution, metrics, or DB write fails.
- Do not swallow exceptions silently.
- If exception occurs after the position was removed, do not re-add it.

### 3. Separate Partial TP From Full Close Lock
`PARTIAL_TP_25` must not be treated as full position close unless remaining size becomes zero.

Implement:
```python
FULL_CLOSE_TYPES = {
    "TP", "SL", "MICRO_TP", "TRAIL_PROFIT", "SCRATCH_EXIT", "STAGNATION_EXIT",
    "TIMEOUT_PROFIT", "TIMEOUT_LOSS", "TIMEOUT_FLAT", "EARLY_STOP",
    "REPLACED_EXIT", "WALL_EXIT", "EMERGENCY_EXIT"
}
PARTIAL_CLOSE_TYPES = {"PARTIAL_TP_25", "PARTIAL_TP_50", "PARTIAL_TP_75"}
```

Rules:
- Partial TP should use a separate partial lock key:
```python
PARTIAL:{symbol}:{side}:{entry}:{exit_type}:{partial_index_or_pct}
```
- Partial TP must update `realized_pnl`, reduce size, persist state, then release partial lock.
- Partial TP must not call force reconcile on the whole position unless remaining size <= 0.
- Full close lock must only be acquired for full-close exits.

### 4. Fix Reconcile Semantics
Force reconcile is emergency-only.

When `_force_reconcile_stuck_close()` runs:
- Mark position as `force_reconciled`.
- Remove from `_positions` only if it still exists.
- Persist a minimal emergency close record with:
```python
exit_type="FORCE_RECONCILE"
reason=<original reason>
net_pnl=<best available mark-to-market net>
integrity_status="force_reconciled"
```
- Do not count it as normal TP/SCRATCH/STAG win.
- Do not increment normal exit audit counters.
- Add block:
```text
[CLOSE_FORCE_RECONCILE_RECORD] symbol=... net_pnl=... persisted=True/False
```

### 5. Normalize Exit Type Before Validation
Map:
```python
"replaced" -> "REPLACED_EXIT"
"wall_exit" -> "WALL_EXIT"
```

No validator should reject `replaced` once normalized.

### 6. Add Close Health Summary
Every 60s:
```text
[CLOSE_HEALTH] active=<n> oldest=<sec> stale_releases=<n> force_reconciles=<n> recent_closed=<n>
```

If `force_reconciles > 0`, include top keys.
