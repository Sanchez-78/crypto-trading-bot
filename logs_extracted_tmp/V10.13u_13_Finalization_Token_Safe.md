## Production Problem

`V10.13u+12` released stale close locks after TTL, but a position could immediately reacquire the same lock every 20 seconds.

Observed failure mode:

```text
[CLOSE_LOCK_STALE_RELEASE] ... stale_count=13
[POSITION_CLOSE_STUCK] ... action=reconcile_required
[CLOSE_LOCK_ACQUIRED] ...
[CLOSE_SKIP_DUPLICATE] ... already_closing age=15s attempts=1000+
```

This created an infinite close loop:

1. position enters close path
2. close lock remains active
3. duplicate close attempts spam logs
4. TTL releases lock
5. same stuck position reacquires lock
6. position never leaves `_positions`

## Implemented Fix

### 1. Force-reconcile thresholds

Added constants in `src/services/trade_executor.py`:

```python
CLOSE_LOCK_FORCE_RECONCILE_AFTER = 3
CLOSE_LOCK_MAX_ATTEMPTS = 250
CLOSE_DUP_LOG_INTERVAL_S = 10.0
```

Meaning:

- after 3 stale releases, force reconcile the stuck position
- or after 250 duplicate close attempts, force reconcile
- duplicate close logs are throttled to 10 seconds

### 2. New helper: `_force_reconcile_stuck_close()`

Behavior:

- removes stuck position from `_positions` under `_positions_lock`
- marks close key as `force_reconciled` in `_RECENTLY_CLOSED`
- blocks immediate reacquire for `RECENTLY_CLOSED_TTL_S = 60s`
- logs:

```text
[CLOSE_FORCE_RECONCILE] key=... symbol=... stale_count=... attempts=... action=removed_position
```

### 3. Updated `_release_stale_close_lock()`

Now:

- increments stale count
- checks stale count and attempts
- calls `_force_reconcile_stuck_close()` when thresholds are exceeded
- prevents the same key from immediately reacquiring after force reconcile

### 4. Refactored `_cleanup_close_locks()`

Now all stale lock cleanup routes through `_release_stale_close_lock()`.

This avoids duplicate stale-release logic and ensures force reconcile is applied consistently.

### 5. Updated duplicate log throttling

`_try_acquire_close_lock()` now uses:

```python
CLOSE_DUP_LOG_INTERVAL_S = 10.0
```

instead of a hardcoded 5-second interval.
