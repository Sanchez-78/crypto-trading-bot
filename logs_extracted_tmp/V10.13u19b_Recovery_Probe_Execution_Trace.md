# V10.13u+19b — Recovery Probe Execution Trace + Ready-But-Rejected Fix

## Context

Production is running CryptoMaster commit `2b5a77d` (`V10.13u+19`). ECON BAD diagnostics are visible and canonical PF is correct:

```text
pf=0.739 econ_status=BAD pf_source=lm_economic_health pf_fallback=false
```

Live logs repeatedly show:

```text
best_symbol=ADAUSDT best_ev=0.0434 best_score=0.204 best_p=0.523 best_coh=0.868 best_af=0.750 probe_ready=True probe_block=none
```

But there is no matching:

```text
[ECON_BAD_RECOVERY_PROBE]
[ECON_BAD_DEADLOCK_PROBE]
```

and the bot still emits:

```text
decision=REJECT_ECON_BAD_ENTRY weak_ev (ev=0.0300<0.045)
decision=REJECT_ECON_BAD_ENTRY weak_ev (ev=0.0348<0.045)
```

Interpretation:
- This is **not** a threshold-loosening task.
- `EV=0.0434` is above the V10.13u+17 recovery probe floor (`0.038`) and below the normal ECON BAD entry floor (`0.045`).
- V10.13u+19 deadlock probe is only for the `0.0370–0.0380` EV near-miss band, so it is not the right mechanism here.
- If the **current evaluated signal** truly has `probe_ready=True` and `probe_block=none`, it should either open as a controlled recovery probe or log an exact blocker.

## Objective

Implement `V10.13u+19b` to make the recovery probe path auditable and fix the case where a current signal is `probe_ready=True` but still returns `REJECT_ECON_BAD_ENTRY` without an explicit blocker.

## Non-Negotiable Safety Rules

Do **not** loosen:
- negative/zero EV hard block
- ECON BAD normal entry floor (`0.045`)
- V10.13u+17 recovery probe floors
- V10.13u+19 deadlock probe floors
- open-position cap
- cooldown / max probes per hour / max probes per 24h
- forbidden tags: `LOSS_CLUSTER`, `TOXIC`, `SPREAD`, `NEGATIVE_EV`, `FAST_FAIL`, etc.
- forced weak/exploration block
- close-lock / execution integrity rules

Do **not** add Firebase writes.  
Do **not** change TP/SL/exit logic.  
Do **not** use stale `best_seen_*` diagnostics to open a trade.  
A trade may open only from the **currently evaluated signal**.

## Required Implementation

### 1. Separate current-signal readiness from best-seen diagnostics

In `src/services/realtime_decision_engine.py`, distinguish:

```text
best_seen_symbol=...
best_seen_ev=...
```

from:

```text
current_symbol=...
current_ev=...
current_score=...
current_p=...
current_coh=...
current_af=...
current_probe_ready=...
current_probe_block=...
```

The existing `best_symbol/best_ev` snapshot may be historical. It must not imply the current signal is tradable.

### 2. Add recovery probe trace helper

Add a helper such as:

```python
def _trace_econ_bad_recovery_decision(
    *,
    symbol: str,
    ev: float,
    score: float,
    p: float,
    coh: float,
    af: float,
    econ_status: str,
    pf: float,
    current_probe_ready: bool,
    current_probe_block: str,
    final_decision: str,
    reason: str,
    idle_s: float | None = None,
    open_positions: int | None = None,
    cooldown_left_s: float | None = None,
    probes_last_hour: int | None = None,
    probes_last_24h: int | None = None,
    forced: bool = False,
    forbidden_tags: list[str] | None = None,
) -> None:
    """WARNING-level observability only. Never raises. No Firebase writes."""
```

It should emit:

```text
[ECON_BAD_RECOVERY_TRACE] symbol=ADAUSDT ev=0.0434 score=0.204 p=0.523 coh=0.868 af=0.750 pf=0.739 econ_status=BAD current_probe_ready=True current_probe_block=none final_decision=TAKE|REJECT reason=...
```

Throttle if needed, but do not suppress `current_probe_ready=True final_decision=REJECT`.

### 3. Add ready-but-rejected invariant log

If the current signal evaluates as:

```text
current_probe_ready=True current_probe_block=none
```

but the function returns `REJECT_ECON_BAD_ENTRY`, emit:

```text
[ECON_BAD_READY_BUT_REJECTED] symbol=... ev=... score=... p=... coh=... af=... pf=... reason=...
```

The `reason` must be exact, for example:

```text
reason=open_position_cap
reason=probe_cooldown
reason=max_probes_24h
reason=execution_gate
reason=state_idle_unknown
reason=return_path_order_bug
```

If no blocker exists, this is a bug and the signal should become a recovery probe `TAKE`.

### 4. Fix return-path ordering if needed

Find the V10.13u+17 recovery probe path inside `evaluate_signal()`.

Expected behavior:
- Normal ECON BAD entry still blocks below `0.045`.
- Before returning `REJECT_ECON_BAD_ENTRY weak_ev`, check V10.13u+17 recovery probe allowance for the **current signal**.
- If current signal passes recovery probe rules:
  - set metadata:
    ```python
    signal["_econ_bad_recovery_probe"] = True
    signal["_econ_bad_probe_size_mult"] = 0.15
    signal["_econ_bad_probe_reason"] = "controlled_recovery_probe"
    ```
  - emit:
    ```text
    [ECON_BAD_RECOVERY_PROBE] symbol=ADAUSDT ev=0.0434 score=0.204 p=0.523 coh=0.868 af=0.750 pf=0.739 size_mult=0.15 reason=controlled_recovery_probe
    ```
  - return `TAKE` using the existing decision format.
- If current signal does not pass recovery probe rules:
  - return reject as before
  - emit trace with exact blocker.

Do not use best-seen diagnostic candidate to open trades.


Acceptable outcomes:

### A. Probe opens

```text
[ECON_BAD_RECOVERY_PROBE] symbol=ADAUSDT ev=0.0434 ... size_mult=0.15
```

### B. Probe does not open, but exact blocker is visible

```text
[ECON_BAD_READY_BUT_REJECTED] symbol=ADAUSDT ... reason=probe_cooldown
```

### C. Bug remains

If logs continue showing:

```text
probe_ready=True probe_block=none
decision=REJECT_ECON_BAD_ENTRY
```

without `[ECON_BAD_READY_BUT_REJECTED]` or `[ECON_BAD_RECOVERY_TRACE]`, the trace is not wired into the real production return path.

## Success Criteria

- No weak junk trades.
- No negative EV trades.
- No Firebase quota impact.
- No exit logic changes.
- If current signal is recovery-ready, it either:
  - opens as `[ECON_BAD_RECOVERY_PROBE]`, or
  - logs a single exact blocker.
- No repeated `probe_ready=True probe_block=none` with silent rejection.
