# V10.13u+4 — Canonical Economic PF Real Source Fix

## Production symptom

Live logs still show source drift:

```text
Dashboard Profit Factor: 0.75x
Economic PF: 12.08
Economic: 0.874 [GOOD]
[ECON_CANONICAL_ACTIVE] pf=12.08 source=canonical_profit_factor economic_score=0.874 status=GOOD
```

This means `learning_monitor.py` now calls a function named `canonical_profit_factor`, but that function still reads a stale/non-dashboard dataset when called without explicit closed trades.

## Root cause

`src.services.canonical_metrics.canonical_profit_factor()` is not using the same authoritative closed-trades snapshot as the dashboard.

Likely failure mode:
- Dashboard calculates PF from `canonical_state` / latest 500 closed trades.
- Economic health calls `canonical_profit_factor()` with `closed_trades=None`.
- `canonical_profit_factor()` falls back to stale/global/model-state data and returns `12.08`.
- Diagnostic log is misleading because it says `source=canonical_profit_factor`, but not which trade source/count was used.

## Safety requirement

Economic Health must never report GOOD when dashboard canonical PF is < 1.0 and closed net PnL <= 0.

Expected after fix:

```text
Profit Factor 0.75x
[ECON_CANONICAL_ACTIVE] pf=0.75 source=canonical_dashboard closed_trades=500 ...
Economic: ... [BAD|CAUTION|FRAGILE]
```

Forbidden:

```text
PF: 12.08
Economic: ... [GOOD]
[ECON_CANONICAL_ACTIVE] pf=12.08
```

## Implementation task

You are working in existing CryptoMaster repository. Apply the smallest safe patch. Do not rewrite architecture.

### 1. Find dashboard PF source

Search:

```bash
grep -R "Profit Factor" -n src bot2 main.py start.py
grep -R "canonical_profit_factor\|profit_factor\|canonical_state\|closed_trades" -n src bot2 main.py start.py
```

Identify the exact function/data object used to print:

```text
Profit Factor  0.75x
```

That source is authoritative.

### 2. Change canonical_metrics.py to accept explicit source and expose diagnostics

In `src/services/canonical_metrics.py`:

- `canonical_profit_factor(closed_trades=None)` must:
  - If `closed_trades` is provided: compute only from that list.
  - If not provided: read from the same authoritative canonical closed-trades source as dashboard.
  - Never silently fall back to stale LM/model-state/Redis PF.
  - Return `0.0` or `float("inf")` only by explicit documented rules.
  - Add a diagnostic helper, for example:

```python
def canonical_profit_factor_with_meta(closed_trades=None) -> dict:
    return {
        "pf": pf,
        "source": source_name,
        "closed_trades": len(trades),
        "wins": wins,
        "losses": losses,
        "gross_win": gross_win,
        "gross_loss": gross_loss,
        "net_pnl": net_pnl,
    }
```

### 3. Change learning_monitor.py to use meta result

In `src/services/learning_monitor.py`, inside economic health calculation:

- Replace blind call:

```python
pf = canonical_profit_factor()
```

with explicit meta call:

```python
meta = canonical_profit_factor_with_meta(canonical_closed_trades_or_dashboard_source)
pf = meta["pf"]
net_pnl = meta["net_pnl"]
```

If `learning_monitor.py` does not have direct access to canonical closed trades, import the same canonical-state getter used by dashboard. Do not use LM-local trades as fallback unless explicitly logged as fallback and marked unsafe.

### 4. Hard safety clamp

After PF/net_pnl calculation:

```python
if pf < 1.0 and net_pnl <= 0:
    economic_score = min(economic_score, 0.34)
    status = "BAD"
elif pf < 1.0:
    economic_score = min(economic_score, 0.49)
    status = "CAUTION"
```

This must run after all score calculation and before logging/return.
