# V10.13m — Exit Attribution Audit Patch

## CONTEXT

Current live state after V10.13k + V10.13L:

- `scratch` counters are finally visible and firing
- `micro` counters are visible
- runtime fault registry is working
- bot is stable again after restart/sync issues
- however, exit mix is still poor:
  - `t_flat` remains dominant
  - `TP` is near zero
  - `trail` is near zero
  - `harvest` is near zero or negligible

This means the next step must **not** be blind threshold tuning.

The system first needs **high-resolution exit attribution telemetry** so we can prove why a position became:
- `SCRATCH_EXIT`
- `MICRO_TP`
- `PARTIAL_TP_*`
- `TRAIL_*`
- `TIMEOUT_FLAT`
- `TIMEOUT_PROFIT`
- `TIMEOUT_LOSS`
- `SL`

Right now, we can see final counters, but not the precise path that caused one exit branch to win over another.

---

## OBJECTIVE

Implement a **diagnostic-only patch** that explains, for every evaluated open position:

1. which exit branches were checked
2. what the relevant numeric inputs were
3. why each branch did or did not trigger
4. which branch finally won
5. whether timeout is structurally pre-empting better exits

This patch must be:
- **safe**
- **non-invasive**
- **toggleable**
- **low-overhead**
- **production-usable for short observation windows**

Do **not** redesign exit logic yet.  
Do **not** change thresholds yet unless required for safety.  
This patch is for **observability first**.

---

## PRIMARY HYPOTHESIS TO TEST

We need to verify which of these is true:

### Hypothesis A — Scratch is still too narrow
Scratch is reachable now, but its trigger window is too strict, so most near-flat positions still age into timeout.

### Hypothesis B — Timeout wins too early
Timeout classification may still be capturing positions before scratch / harvest / trail can trigger.

### Hypothesis C — Trailing never arms
Trailing activation may be too late, or MFE/retrace logic may be broken or under-tracked.

### Hypothesis D — Micro / partial TP thresholds are mismatched to real move distribution
Signals may generate small favorable excursions, but not enough to cross current levels.

### Hypothesis E — Exit ordering is correct, but numeric state is wrong
Example:
- bad `mfe`
- bad `tp_progress`
- bad age
- stale regime
- wrong unrealized PnL basis
- missing price normalization

This patch must generate evidence to confirm which hypothesis is actually true.

## IMPLEMENTATION SCOPE

### 1. Add explicit exit attribution telemetry in `smart_exit_engine.py`

Create a structured debug helper, for example:

```python
def _log_exit_eval(position, branch, decision, **kwargs): ...
```

It should log only when a debug flag is enabled.

Recommended fields:

- `sym`
- `side`
- `regime`
- `age_s`
- `entry`
- `curr`
- `pnl_pct`
- `tp_pct`
- `sl_pct`
- `tp_progress`
- `mfe_pct`
- `mae_pct`
- `branch`
- `decision`
- `reason`
- `threshold`
- `observed`
- `retrace_pct` if trailing relevant
- `timeout_s_remaining` if relevant

#### Required branch coverage
For each evaluated position, log PASS / FAIL for:

- `MICRO_TP`
- `BREAKEVEN_STOP`
- `PARTIAL_TP_25`
- `PARTIAL_TP_50`
- `PARTIAL_TP_75`
- `TRAILING_STOP`
- `SCRATCH_EXIT`
- `STAGNATION_EXIT`
- `TIMEOUT`

Do not only log the winner.  
Log enough to explain **why non-winning branches failed**.

Example style:

```python
[EXIT_AUDIT] BTCUSDT BUY age=127 pnl=0.00018 tp_prog=0.21 mfe=0.00024 branch=SCRATCH_EXIT FAIL reason=pnl_above_scratch_cap observed=0.00018 threshold=0.00010
[EXIT_AUDIT] BTCUSDT BUY age=127 pnl=0.00018 tp_prog=0.21 mfe=0.00024 branch=TIMEOUT FAIL reason=age_below_timeout observed=127 threshold=180
[EXIT_AUDIT] BTCUSDT BUY age=192 pnl=0.00003 tp_prog=0.05 mfe=0.00026 branch=SCRATCH_EXIT TAKE reason=flat_release_window
```

### 4. Track timeout pre-emption explicitly

We need a dedicated diagnostic for the core question:
**Did timeout win while another exit was almost eligible?**

Add a near-miss tracker:

```python
_timeout_preemptions = {
    "scratch_near_miss": 0,
    "micro_near_miss": 0,
    "trail_near_miss": 0,
    "partial25_near_miss": 0,
}
```

Suggested near-miss definition:
- within 10–20% of threshold, or
- age eligible but price condition narrowly missed, or
- branch armed historically but lost on current tick

Examples:
- timeout at age 300 with scratch pnl band missed by tiny margin
- timeout with trailing armed but retrace not reached
- timeout with micro threshold missed by 0.00001

This tells us whether to retune thresholds, ordering, or state tracking.

- whether `mae_pct` updates correctly
- whether trailing has an `armed` state
