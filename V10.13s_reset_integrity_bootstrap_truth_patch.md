# V10.13s — Reset Integrity + Bootstrap Truth Patch

## Role
You are a senior quant/backend engineer working directly in the existing CryptoMaster Python codebase.

Your task is to fix a **state integrity bug after DB wipe / restart**.
The current live logs show that the bot is running, trading, and bootstrap softening is partially working — but its internal state is contradictory.

This patch is about **making bootstrap state truthful and consistent**.

---

## Critical Live Evidence

Recent live logs show these contradictions at the same time:

### Global state says:
- `completed_trades: 2306`
- dashboard says:
  - `Kalibrace KALIBROVAN ✓`
  - `AKTIVNI – robot je kalibrovany a obchoduje!`
- auditor threshold is high:
  - `EV prah 0.142`
  - `score prah 0.173`

### But pair/regime state says:
- per pair/regime counts are tiny:
  - many entries only `n=3..6`
- feature WR values are tiny / bootstrap-like:
  - around `0.2453`
- learning health is effectively dead:
  - `Health: 0.000 [BAD]`
  - `LEARNING: NO LEARNING SIGNAL DETECTED`

### Additional proof
Cold-start logs are clearly active:
- `[V10.13r] SCORE_THRESHOLD_COLD_START: relaxed ...`
- `OFI_TOXIC_SOFT_BOOTSTRAP`

So the bot is simultaneously:
- in bootstrap
- and also acting like it is fully calibrated

That state is invalid.

---

## Root Cause Interpretation

This is almost certainly a **partial reset / stale warm-start contamination** problem.

Most likely one or more of these are true:

1. Global metrics such as:
- `completed_trades`
- `equity_peak`
- calibration flags
- mature auditor thresholds
- historical performance summaries

are being restored from stale persisted state,

while at the same time:

2. Pair/regime learning state has been reset or partially reset.

That creates a mixed state:
- mature global thresholds
- immature local learning
- bootstrap rescue partially active
- but decision system still anchored to old global totals

This distorts:
- EV thresholding
- score thresholding
- calibration status
- dashboard mode
- recovery logic
- confidence in learning monitor

---

## Objective

Implement a **Reset Integrity + Bootstrap Truth Patch** so that after DB wipe / restart the bot cannot enter a mixed state.

When the learning state is fresh, **all derived global status must also behave fresh**.

---

## Required Fixes

## 1. Add state consistency validation on startup
Create a deterministic validation function, for example:

- `validate_runtime_state_consistency()`
- or `detect_stale_warm_start_state()`

It should run during startup / warm start hydration and compare:

- global completed trades
- global calibration / mode flags
- pair/regime counts
- feature history size
- persisted metrics
- recent trade history availability

### Required behavior
If the bot finds a mismatch like:
- global `completed_trades` very high
- but median / max pair `n` very low
- and learning monitor clearly has sparse state

then it must detect:
- `STATE_MISMATCH_RESET_REQUIRED`
or similar

This must be explicit in logs.

---

## 2. Recompute completed_trades from trustworthy source
Do **not** blindly trust stale global counters after wipe.

### Required behavior
At startup, rebuild or validate `completed_trades` from the actual trustworthy source, in priority order such as:
1. live trade history collection / persisted trades
2. learning monitor pair counts if that is safer
3. fallback to 0 if state is sparse / inconsistent

### Important
If trade DB was wiped, `completed_trades` must not remain at old values like 2306.

After wipe, a realistic value should be near actual fresh sample count — not old historical total.

---

## 3. Force calibration status to follow real learning maturity
Current problem:
- dashboard says calibrated even though state is clearly bootstrap

Required patch:
Calibration / active mode must be based on **real current maturity**, not stale historical flags.

### Required rule
The bot can only show:
- calibrated
- active / mature

if current state genuinely supports it.

Use logic based on:
- enough recent completed trades
- enough pair/regime sample depth
- enough feature history
- non-contradictory state

Otherwise show:
- bootstrap
- training
- collecting data
- or similar truthful status

No fake “calibrated” state after reset.

---

## 4. Auditor thresholds must use effective sample size, not stale sample size
Current problem from logs:
- `EV prah 0.142` even though pair Ns are only 3–6
- this is too mature for fresh bootstrap

Required patch:
Introduce an **effective learning sample size** for threshold logic.

### Example concept
Instead of using only stale global completed_trades:
- derive `effective_n` from current reliable state
- cap mature thresholds when pair/regime maturity is still low
- if state mismatch exists, bias to bootstrap-safe thresholds

### Goal
After wipe:
- EV threshold should behave like early bootstrap
- not like a 2300-trade mature system

---

When startup consistency check detects stale warm-start contamination, automatically reset or invalidate these kinds of fields as appropriate:

### Important
Do this surgically.
Do not destroy valid live state if it genuinely exists.
But when wipe/reset happened, stale maturity must be cleared.
