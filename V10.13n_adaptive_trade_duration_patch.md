# V10.13n — Adaptive Trade Duration Manager

**Status:** návrh patche  
**Cíl:** omezit předčasné uzavírání obchodů a současně nenechávat mrtvé obchody viset příliš dlouho  
**Typ:** bezpečný behavior patch nad exit vrstvou  

---

## Problém

Z aktuálních logů a předchozích patchů je vidět, že:

- některé obchody se zavírají příliš brzy,
- exit mix je stále slabý (`TP=0`, `trail=0`, harvest nízký),
- velká část zavření jde přes timeout / scratch-like logiku,
- pevné časové hranice (`SCRATCH_MIN_AGE`, timeouty, stagnation) jsou příliš statické pro různé režimy trhu.

To znamená, že systém používá **jednu délku života obchodu pro různé typy situací**, což je strukturálně špatně.

---

## Cíl patche

Zavést **automatické, metrikami řízené nastavování délky obchodu**.

Jinými slovy:

- silný trendový obchod má dostat více času,
- slabý / flat / noisy obchod má být ukončen dřív,
- obchod s rostoucím MFE má dostat čas „dodejchat“ profit,
- obchod bez follow-through má být zkrácen,
- timeout, scratch a stagnation musí být **adaptivní**, ne fixní.

---

## Hlavní návrh: Adaptive Trade Duration Manager (ATDM)

Nový modul například:

`src/services/adaptive_trade_duration.py`

Bude vracet pro každý otevřený obchod dynamický profil:

```python
{
    "min_hold_s": 90,
    "scratch_enable_s": 110,
    "stagnation_enable_s": 180,
    "timeout_s": 420,
    "trail_activation_mult": 0.85,
    "harvest_delay_mult": 1.10,
}
```

Tyto hodnoty nebudou pevné, ale spočítané z metrik.

## Metriky, podle kterých to řídit

### 1. Režim trhu

Použít existující regime klasifikaci:

- `BULL_TREND`
- `BEAR_TREND`
- `RANGING`
- `QUIET_RANGE`
- `HIGH_VOL`

Princip:

- **trend** → delší držení
- **range / quiet** → kratší držení
- **high vol** → rychlejší reakce, ale ne nutně kratší timeout; spíš dřívější trail a harvest

Příklad:

```python
REGIME_TIMEOUT_MULT = {
    "BULL_TREND": 1.35,
    "BEAR_TREND": 1.25,
    "RANGING": 0.85,
    "QUIET_RANGE": 0.70,
    "HIGH_VOL": 0.95,
}
```

---

### 2. Kvalita vstupu při otevření

Použít signálové metriky:

- `score`
- `coherence`
- `ws`
- `p`
- `ev`
- počet potvrzených features

Silnější setup má dostat více času.

Příklad:

```python
entry_quality = (
    0.30 * normalized_score +
    0.25 * coherence +
    0.20 * ws +
    0.15 * p +
    0.10 * feature_confirm_ratio
)
```

Interpretace:

- `entry_quality >= 0.75` → prodluž hold
- `0.55 - 0.75` → neutrální
- `< 0.55` → zkrať hold

---

### 3. Follow-through po vstupu

Tohle je nejdůležitější pro tvůj problém.

Po otevření obchodu měřit:

- `MFE` (max favorable excursion)
- `MAE` (max adverse excursion)
- čas do prvního plusu
- čas do dosažení 25 % TP progresu
- poměr `MFE / age`
- jestli cena drží směr nebo se vrací do entry

Pokud obchod rychle ukáže follow-through, má dostat více času.
Pokud jen stojí na místě, má být zkrácen.

Příklad pravidla:

```python
if age_s >= 60 and tp_progress < 0.10 and mfe_pct < 0.00015:
    timeout_mult *= 0.80
    stagnation_enable_s *= 0.75
```

Naopak:

```python
if tp_progress >= 0.35 or mfe_pct >= 0.00040:
    timeout_mult *= 1.25
    scratch_enable_s *= 1.20
```

---

### 4. Market microstructure / execution quality

Už máš:

- OFI
- spread
- execution quality
- cost guard

Když je prostředí horší, je lepší obchod držet kratší dobu, protože edge se rychleji rozpadá.

Příklad:

```python
if exec_quality < 0.55 or spread_ratio > spread_warn:
    timeout_mult *= 0.85
    trail_activation_mult *= 0.90
```

---

### 5. Pair-regime learning stats

Použij live statistiky páru v daném režimu:

- `wr`
- `ev`
- `conv`
- `bandit`
- `n`

Když má pár+režim dobrou historii, nech obchodům víc prostoru.
Když je pár v režimu slabý, nečekej zbytečně dlouho.

Příklad:

```python
if n >= 15 and wr >= 0.58 and ev > 0:
    timeout_mult *= 1.15
elif n >= 10 and wr < 0.45:
    timeout_mult *= 0.85
```

## Doporučené chování

### A. Silný trendový winner

Vstup:

- režim `BULL_TREND`
- vysoké `score`, `coherence`, `ws`
- rychlý MFE po vstupu

Chování:

- delší `timeout_s`
- pozdější `scratch_enable_s`
- trail může být dříve aktivní, ale ne příliš agresivní
- harvest spíš později než u slabého setupu

Výsledek: obchod se nezavře předčasně.

---

### B. Flat / nudný obchod

Vstup:

- `RANGING` nebo `QUIET_RANGE`
- průměrná kvalita
- dlouho bez MFE

Chování:

- kratší timeout
- dřívější scratch
- dřívější stagnation exit

Výsledek: kapitál se uvolní dřív.

---

### C. Volatilní obchod s rychlým ziskem

Vstup:

- `HIGH_VOL`
- rychlý MFE
- vyšší spread / noise

Chování:

- ne nutně kratší celý timeout
- ale dřívější trail / partial harvest
- méně prostoru pro návrat profitu zpět

Výsledek: profit se neztratí.

## Nejlepší minimální implementace

Aby patch byl bezpečný a inkrementální, doporučuji 3 fáze.

### Fáze 1 — pouze timeout/scratch/stagnation adaptace

Bez zásahu do vstupu a sizingu.

Přidat:

- `get_trade_duration_profile(position, market_state, learning_state)`
- napojit do `smart_exit_engine.evaluate_position_exit()`
- nahradit konstanty dynamickými hodnotami

To samo by mělo vyřešit velkou část problému „zavírá se moc brzo“.

---

### Fáze 2 — audit telemetry

Logovat pro každý uzavřený obchod:

```python
[ATDM] BTCUSDT BULL_TREND age=134s timeout=420 scratch_at=120 stagnation_at=240
       quality=0.78 follow=0.66 regime_mult=1.35 learning_mult=1.12 final_mult=1.41
```

A pro near-miss případy:

```python
[ATDM_NEAR] ETHUSDT scratch prevented by strong_followthrough
```

---

### Fáze 3 — online self-tuning

Po nasbírání dat můžeš adaptovat i samotné multiplikátory:

- pokud obchody s delším holdem mají lepší PF → posuň timeout výš pro daný režim,
- pokud delší hold jen zvyšuje timeout losses → timeout dolů,
- pokud scratch exits často předcházejí pozdějším winnerům → scratch odložit,
- pokud scratch exits zachraňují flat obchody → scratch držet dříve.

To už je ale až V10.13o/p.

---

## Co bych upravil hned teď

Podle toho, co píšeš, bych dal tyto priority:

### Patch 1
**Adaptive timeout / scratch / stagnation podle režimu + follow-through**

To je nejvyšší priorita, protože přímo řeší předčasné zavírání.

### Patch 2
**Delayed scratch for strong setups**

Když:

- vysoké `score`
- vysoká `coherence`
- rychle dosažené pozitivní MFE

pak scratch odložit.

### Patch 3
**Fast kill pro mrtvé obchody**

Když po X sekundách není žádný follow-through, ukončit dřív.

To zlepší turnover bez poškození winnerů.

## Proč je to lepší než pevný timeout

Pevný timeout říká:

> každý obchod má stejnou šanci potřebovat stejný čas

což není pravda.

Adaptivní timeout říká:

> obchod dostane tolik času, kolik si zaslouží podle kvality, režimu a reálného vývoje po vstupu

To je přesně to, co chceš, pokud máš pocit, že se některé obchody zavírají moc brzo.

## Rizika

### Riziko 1 — příliš dlouhé držení špatných obchodů
Řešení:
- horní limit timeoutu,
- follow-through multiplier nesmí jen prodlužovat, ale i zkracovat,
- stagnation musí zůstat aktivní.

### Riziko 2 — příliš složitá logika
Řešení:
- první verzi udělat jen přes 4 multiplikátory,
- auditovat a až potom rozšiřovat.

### Riziko 3 — zmatek v observabilitě
Řešení:
- přidat `[ATDM]` telemetry snapshot do dashboardu a exit audit sekce.

## Očekávaný efekt

Pokud je hypotéza správná, po patchi bys měl vidět:

- méně předčasných scratch/timeout-flat exitů u silných setupů,
- vyšší podíl micro/partial/trail/harvest exitů,
- lepší PF,
- nižší frustraci z obchodů, které se zavřou a pak jdou správným směrem.
