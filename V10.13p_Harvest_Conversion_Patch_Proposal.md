# V10.13p — Harvest Conversion Patch Proposal

## Current Assessment

V10.13o is **stable and profitable**, but it still does **not fully solve exit distribution**.

### What is working
- Bot is running normally
- Trades continue without stalls
- Winrate is healthy: **54.8%**
- Profit Factor is strong: **4.02x**
- Expectancy is positive
- Timeouts are effectively eliminated
- `PARTIAL_TP_25` is now firing: `partial=(8,0,0)`
- Harvest rate improved to **11.8% (13/110)**

### Main remaining issue
Exit flow is still dominated by **SCRATCH_EXIT**:

- `scratch=97`
- `partial25=8`
- `micro=4`
- `trail=1`
- `partial50=0`
- `partial75=0`

This means most positions are still being released early instead of maturing into higher-value exits.

---

## Evidence From Live Logs

### Exit line
```text
[V10.13g EXIT] TP=0 SL=0 micro=4 be=0 partial=(8,0,0) trail=1 scratch=97 stag=0 harvest=0 t_profit=0 t_flat=0 t_loss=0
```

### Exit audit
```text
winners: SCRATCH_EXIT=32 PARTIAL_TP_25=7 MICRO_TP=1
near_miss: partial25_near_miss=27804 micro_near_miss=15865 scratch_near_miss=818
```

### Key interpretation
This is the strongest signal in the whole dataset:

- `partial25_near_miss=27804` is extremely high
- `micro_near_miss=15865` is also very high
- `scratch_near_miss=818` is much lower

This strongly suggests:

1. trades often get **very close** to `PARTIAL_TP_25`
2. they still fail to convert often enough
3. the system is **not far away** from a better harvest mix
4. the next patch should focus on **converting near-misses into real partial/trailing exits**

---

## Secondary Problem: Entry Pipeline Still Too Restrictive

The system is now stable on exits, but the entry side is still heavily constrained.

### Observed block reasons
- `FREQ_CAP = 991`
- `OFI_TOXIC_HARD = 265`
- `FAST_FAIL_HARD = 103`
- `FAST_FAIL_SOFT = 189`

This is a lot of blocking. Even a better exit engine cannot fully express itself if too many viable entries are being filtered out upstream.

---

## Weak Pair / Regime Segments

### Clear underperformers
- `SOLUSDT_BEAR_TREND` → **19% WR**
- `DOTUSDT_BEAR_TREND` → **31% WR**
- `BTCUSDT_BEAR_TREND` → **37.5% WR**
- `ETHUSDT_BEAR_TREND` → **36.8% WR** despite slightly positive EV

### More acceptable / stronger
- ETH overall
- BNB
- XRP is inconclusive due to low sample size

This means the next patch should **not only adjust exits**, but also apply **pair/regime protection** where evidence is already strong.

---

## Conclusion

### V10.13o verdict
**PASS** on:
- runtime stability
- no timeout regression
- profitability
- exit logic functioning

**FAIL / incomplete** on:
- true exit diversification
- promotion from scratch to trailing / higher partial layers
- reducing scratch dominance enough

### Strategic conclusion
Do **not** revert V10.13o.

Instead, build the next patch specifically around:

- converting `partial25_near_miss` into actual `PARTIAL_TP_25`
- giving trailing a slightly better chance to arm
- avoiding another oversized scratch adjustment that could damage PF

# V10.13p — Proposed Patch Direction

## Objective
Increase harvest conversion **without destabilizing PF or reintroducing timeouts**.

---

## Recommended Changes

### 1. Lightly reduce `PARTIAL_TP_25`
Reason:
- extremely high `partial25_near_miss`
- evidence says threshold is still just slightly too far away

Guideline:
- reduce only a little
- do **not** make a large jump
- goal is conversion, not aggressive premature harvesting

Expected effect:
- more real `PARTIAL_TP_25`
- some trades that currently end as scratch should convert earlier into structured harvesting

---

### 2. Lower `TRAILING_ACTIVATION` slightly again
Reason:
- `trail=1` is still effectively dead
- current tuning improved partials more than trailing
- trailing needs a bit more access to armed state

Guideline:
- small reduction only
- no broad rewrite
- preserve current PF advantage

Expected effect:
- more positions transition from partial profit state into a trailing-managed finish
- reduced reliance on scratch

---

### 3. Leave scratch mostly unchanged
Reason:
- V10.13o already adjusted scratch in the correct direction
- PF is currently very strong
- another large scratch shift is more likely to create instability than benefit

Guideline:
- avoid major new changes to:
  - `SCRATCH_MIN_AGE_S`
  - `SCRATCH_MAX_PNL`

Optional:
- only micro-adjust if post-patch evidence still shows scratch >70% after enough exits

Expected effect:
- preserve current safety and PF
- let partial/trailing improvements do the real work

---

### 4. Add stronger pair/regime suppression for weak segments
Reason:
Live evidence now clearly shows repeated weak areas.

Highest-priority candidates:
- `SOLUSDT_BEAR_TREND`
- `DOTUSDT_BEAR_TREND`

Optional watchlist:
- `BTCUSDT_BEAR_TREND`
- `ETHUSDT_BEAR_TREND`

Recommended action:
- not a full hard block
- use stronger size suppression and/or stricter local auditor factor
- optionally add temporary local cooldown behavior for very weak combinations

Expected effect:
- reduce drag from proven weak segments
- preserve learning while cutting cost of repeated bad exposures

---

### 5. Review `FREQ_CAP`
Reason:
- `FREQ_CAP=991` is extremely high
- may be choking too many valid follow-up entries
- entry starvation can indirectly distort exit mix by lowering the sample of better-quality developing trades

Recommended action:
- do not immediately loosen it blindly
- first instrument or review where it fires most
- determine whether it is suppressing low-quality spam or also suppressing quality continuation entries

Expected effect:
- possible future improvement in trade quality and flow
- should be a controlled follow-up, not the main change of V10.13p unless logs already prove overblocking

---

# Final Recommendation

Proceed with **V10.13p as a harvest-conversion patch**, not as another scratch-dominance patch.

That means:

- **reduce `PARTIAL_TP_25` slightly**
- **reduce `TRAILING_ACTIVATION` slightly**
- **avoid large new scratch changes**
- **penalize SOL/DOT BEAR segments more strongly**
- **treat `FREQ_CAP` as a separately instrumented bottleneck**

This is the highest-probability next step based on the current evidence.
