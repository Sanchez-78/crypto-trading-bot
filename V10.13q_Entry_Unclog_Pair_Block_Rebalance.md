# V10.13q — Entry Unclog / Pair-Block Rebalance Patch

## Goal

The current system is **stable and profitable overall**, but the **entry pipeline is severely overblocked**.  
The main problem is **not exits anymore**. The main problem is that the bot almost never lets new trades through.

This patch should focus on:

1. Reducing **hard entry deadlock**
2. Converting overly aggressive **pair/regime hard blocks** into staged penalties
3. Making **emergency/idle recovery** actually able to restore flow
4. Fixing the remaining **idle-time anomaly source**
5. Improving **kill-reason observability** so the next patch is data-driven

---

Emergency mode activates, thresholds relax, and some candidates internally reach `decision=TAKE`, but they are still later killed by pair-level hard blocks.

So the system has a contradiction:

- recovery layer says **"allow something through"**
- pair blocker says **"kill almost everything anyway"**

This makes recovery ineffective.

## Root Cause Summary

### Root cause A — `SKIP_PAIR` is too strong
Pair/regime WR-based blocking is no longer a protective filter.  
It has become a **global entry choke point**.

Examples from logs:
- `SKIP_PAIR BTCUSDT/BEAR_TREND wr=29% n=34`
- `SKIP_PAIR ETHUSDT/BEAR_TREND wr=24% n=25`
- `SKIP_PAIR BNBUSDT/BEAR_TREND wr=29% n=31`
- `SKIP_PAIR XRPUSDT/BEAR_TREND wr=12% n=25`

That means the currently dominant market regimes are often rejected wholesale.

### Root cause B — emergency mode cannot override pair-block
Even when:
- idle is large
- positions are zero
- blocked ratio is extreme
- threshold falls to emergency level

…the later `SKIP_PAIR` logic still nullifies the recovery.

### Root cause C — `FAST_FAIL_SOFT` is far too dominant
`FAST_FAIL_SOFT` count is enormous.  
A “soft” decision should usually **degrade quality/size**, not behave like a near-global choke amplifier.

### Root cause D — idle time source is still noisy / partially wrong
The warnings:
- `safe_idle_seconds: unrealistic idle=195222s, resetting`

show that idle-state calculation still uses stale or invalid state in at least some paths.

Even if safety fallback prevents catastrophic behavior, this still:
- pollutes logs
- may trigger false emergency state
- distorts adaptive gating

### Root cause E — observability still stops one layer too early
We have counts of reasons, but we still lack a clean answer to:

**Which final layer killed each candidate?**

Without a final kill attribution summary, tuning remains slower than necessary.

## Required Changes

## 1. Convert `SKIP_PAIR` from hard kill to staged penalty

### Problem
Current behavior is too binary:
- either a pair/regime is tradable
- or it is effectively dead

### Desired behavior
For weak pair/regime combinations:
- reduce size
- increase threshold
- reduce confidence
- only hard block the truly catastrophic ones

### Proposed staged policy

#### Tier 1 — Warning zone
Condition:
- `n >= 15`
- `wr < 0.40`

Action:
- `pair_penalty *= 0.75`
- keep tradable

#### Tier 2 — Strong penalty
Condition:
- `n >= 20`
- `wr < 0.30`

Action:
- `pair_penalty *= 0.50`
- optional extra EV threshold uplift `+0.01`

#### Tier 3 — Severe penalty
Condition:
- `n >= 25`
- `wr < 0.20`

Action:
- `pair_penalty *= 0.25`
- optional score threshold uplift `+0.02`

#### Tier 4 — Hard block
Condition:
- `n >= 30`
- `wr < 0.10`
- and no emergency recovery active

Action:
- hard reject allowed

### Important rule
`SKIP_PAIR` hard block should become **rare**.  
Default behavior should be **degrade, not kill**.

---

## 2. Let emergency mode override pair-block

### Problem
Emergency mode currently lowers thresholds, but downstream hard blocks still prevent execution.

### Required behavior
When all of the following are true:
- `positions == 0`
- idle exceeds threshold
- blocked ratio is high (e.g. `> 0.85`)
- health is weak/bad or recovery mode active

Then pair-level rejection must be softened.

### Proposed emergency override
In emergency mode:

- Tier 1 / Tier 2 / Tier 3 pair states:
  - never hard block
  - convert to penalty only
- Tier 4 hard block:
  - either keep hard block only for catastrophic segments
  - or allow one limited-size exploratory trade every cooldown window

### Suggested exploration fallback
Allow at most:
- `1 exploratory trade / pair / 20–30 min`
- minimum size only
- only if spread / exec quality / portfolio heat are acceptable

This preserves safety while preventing total deadlock.

---

## 3. Rework `FAST_FAIL_SOFT` so it behaves truly soft

### Problem
`FAST_FAIL_SOFT` count is huge, which suggests it acts as a shadow hard filter.

### Required behavior
A soft fail should mostly do one or more of:

- reduce effective EV
- increase score threshold slightly
- reduce size
- lower candidate priority

It should **not** function as a mass kill switch.

### Proposed behavior
Instead of rejecting:
- apply multiplicative confidence penalty
- apply position size penalty
- optionally require stronger downstream confirmation

Example:
- `soft_fail_penalty = 0.55–0.80`
- keep candidate alive unless other guards also fail

### Rule
Only `FAST_FAIL_HARD` should be allowed to kill by itself.  
`FAST_FAIL_SOFT` should almost never be terminal on its own.

---

## 4. Fix idle time source of truth

### Problem
This warning is still appearing repeatedly:
- `safe_idle_seconds: unrealistic idle=195222s, resetting`

This means idle derivation is still contaminated by stale timestamps or mixed time references.

### Required fix
Use a single trusted source for idle:

Preferred order:
1. last actual trade close/open event timestamp
2. last successful execution timestamp
3. last valid portfolio mutation timestamp

Do **not** derive recovery idle from:
- stale boot-time values
- unix/raw state values without sanity bounds
- inconsistent persisted timestamps without validation

### Guardrails
If computed idle is:
- negative
- unrealistically large
- older than process uptime by impossible margin

then:
- clamp safely
- emit warning once per cooldown period
- do not spam the log every cycle

### Expected effect
- cleaner logs
- less false recovery activation
- more stable adaptive gating

---

## 5. Add final kill attribution telemetry

### Problem
We see many counters, but we still do not clearly know the **final terminal cause** for each rejected candidate.

### Required addition
For every candidate, emit/store the final kill reason:
- `PAIR_BLOCK`
- `FAST_FAIL_HARD`
- `FAST_FAIL_SOFT_FINAL`
- `OFI_TOXIC_HARD`
- `FREQ_CAP`
- `SKIP_SCORE_HARD`
- `EXEC_QUALITY`
- `NET_EDGE`
- etc.

### Dashboard summary
Add a compact section such as:

```text
[ENTRY_KILL_AUDIT]
final_kills:
  PAIR_BLOCK=61
  FAST_FAIL_HARD=19
  FREQ_CAP=8
  OFI_TOXIC_HARD=4
  NET_EDGE=2
rescued_by_emergency:
  pair_penalty_only=11
  soft_fail_only=7
```

### Why this matters
This immediately tells us which layer is the real choke point after all upstream penalties are applied.

---

## 6. Add recovery-aware pass-rate target

### Problem
The bot can remain “alive” while effectively non-trading.

### Required behavior
If all of the following happen together:
- positions = 0
- no trade for long period
- audit blocked ratio > target
- emergency mode active

Then the system should enforce a minimum flow objective.

### Suggested targets
- normal mode: pass ratio can be conservative
- recovery mode: require something like:
  - `pass_ratio >= 0.15`
  - or at least `1 accepted candidate per audit window`

If this is not achieved:
- soften pair penalties further
- soften soft-fail further
- temporarily relax score threshold slightly

This creates a controlled anti-deadlock loop.

---

## 7. Keep current exit logic unchanged in this patch

### Important
Do **not** retune exit thresholds in this patch unless absolutely necessary.

Reason:
- exits are no longer the primary bottleneck
- harvest improved materially
- timeout problem is already mostly solved

Changing entry and exit together would make diagnosis harder.

This patch should be **entry-side only**.
